# 校易通从 Demo 到可用 APP 的完整差距清单（全量版）

更新时间：2026-03-18  
适用目标：接入真实数据库 + 真实AI + 可持续迭代 + 可部署上线（校园内测级）

---

## 0. 结论摘要（给项目负责人先看）

当前系统属于 **前端可演示 + 后端 mock 路由** 阶段。  
如果目标是“完整可用 APP（非 Demo）”，不仅是“接口有返回”即可，还必须补齐以下 10 大块：

1. 后端分层与业务服务化（脱离 mock）
2. 数据库建模、迁移、事务与索引
3. 认证鉴权与权限体系（JWT + RBAC）
4. 任务/搭子/信用核心业务规则在后端强校验
5. AI 服务化接入（推荐 + 审核 + 日志闭环）
6. 文件上传与对象存储（图片、学生证）
7. 地图与地理能力后端化（距离计算、附近检索）
8. 安全、审计、风控（限流、输入校验、操作日志）
9. 测试体系（单测/集成/API回归）
10. 部署与运维（环境隔离、配置、监控、备份）

---

## 1. 现状盘点（事实）

## 1.1 后端现状
- 框架：Express
- 路由：`backend/src/routes/*` 基本全部是 mock 返回
- 数据库：未接入（无 ORM/驱动、无迁移）
- 鉴权：无真实 JWT 验签中间件
- 业务：状态机、信用分、权限基本在前端 store 执行
- 测试：无
- 运维：无 Docker、无 CI、无日志系统、无监控

## 1.2 前端现状
- 前端 API 层已定义较完整 endpoint
- 但大量页面仍依赖本地 store 作为主数据源
- AI 推荐/审核目前是前端规则，不是服务
- 地图已接高德前端 SDK，但“附近匹配/地理检索”尚未后端化

## 2. 完整缺口清单（按模块）

## A. 后端工程架构（必须）

### A1. 目录与分层缺口
- [ ] 增加 `controllers/services/repositories/models` 分层
- [ ] 把路由中的业务逻辑迁出到 service
- [ ] 引入 DTO 校验层（如 zod/joi/class-validator）

完成标准：
- 路由文件不直接写业务判断/数据库读写
- 每个接口有明确输入校验和错误码

### A2. 配置与环境缺口
- [ ] 环境变量分层：dev/test/prod
- [ ] 增加配置项：DB、JWT、CORS 白名单、对象存储、AI密钥、高德服务端密钥
- [ ] 增加 `.env.example`（backend）

完成标准：
- 任何敏感配置不硬编码在代码
- 新人可按模板本地启动

### B1. 数据库接入缺口
- [ ] 选型并接入（MySQL + Prisma/Sequelize/TypeORM）
- [ ] 建立迁移机制（migration）
- [ ] 初始化 seed 脚本（测试数据）

完成标准：
- `npm run migrate` 可建全量表

### B2. 核心表缺口（必须落库）

- [ ] `info_posts`
- [ ] `post_reports`
- [ ] `task_posts`
- [ ] `task_applications`（如保留申请制）
- [ ] `task_evaluations`
- [ ] `buddy_posts`
- [ ] `buddy_members`
- [ ] `buddy_evaluations`
- [ ] `buddy_reports`
- [ ] `moderation_logs`
- [ ] `user_behaviors`
- [ ] `admin_operation_logs`

### B3. 约束与事务缺口
- [ ] `credit_score` 强约束 0~100
- [ ] 状态字段做枚举约束（任务/搭子/举报/认证）
- [ ] 唯一约束：手机号、学号（按业务）
- [ ] 任务接单/搭子加入必须事务化（防超卖）

### B4. 索引缺口
- [ ] 列表查询索引：`created_at/status/category`
- [ ] 举报与审核索引：`status, created_at`
- [ ] 地理索引：搭子经纬度（空间索引）

完成标准：
- 常用列表接口在目标数据量下可稳定分页返回

---

## C. 认证、鉴权、权限（必须）

### C1. 认证缺口
- [ ] 用户登录/注册改真实账号体系（密码 hash 或短信验证码）
- [ ] 管理员独立鉴权体系（非前端硬编码）
- [ ] JWT 生成、验签、过期与刷新

### C2. 权限缺口
- [ ] 路由级鉴权中间件
- [ ] 角色权限（user/admin）
- [ ] 资源归属校验（只能改自己的内容）

完成标准：
- 任何敏感接口在无 token、假 token、越权 token 下都被拒绝

---

## D. 业务规则后端化（必须）

### D1. 任务状态机
- [ ] `pending -> ongoing -> pending_confirm -> completed/cancelled` 严格校验
- [ ] 非法跳转返回明确错误码

### D2. 搭子状态机
- [ ] `open/full/completed/cancelled` 后端主导
- [ ] 加入时校验：人数上限、重复加入、半径范围、认证、信用分

### D3. 信用体系
- [ ] 评价映射 `good +2 / neutral 0 / bad -2`
- [ ] 信用变动统一写 `credit_logs`
- [ ] 信用分 < 60 禁止发布/接单/组队加入（后端再校验）

### D4. 举报与审核
- [ ] 举报处理状态统一：`pending/resolved/ignored`
- [ ] 管理员处理必须记录 `admin_note` 和操作日志

完成标准：
- 前端篡改请求参数不能绕过业务规则

---

## E. API 完整性（不仅“有接口”）

## E1. 已有接口需真实化
- [ ] 把当前 mock 路由全部改为真实 DB 读写
- [ ] 统一分页、排序、筛选参数规范
- [ ] 统一错误码规范（参数错/权限错/状态错/系统错）

## E2. 必补接口（按前端当前/近期需要）
- [ ] `PATCH /api/admin/reports/:id/status`
- [ ] `GET /api/tasks/recommendation`
- [ ] `PATCH /api/auth/profile`
- [ ] `POST /api/auth/verification`
- [ ] `GET /api/auth/credit-logs`
- [ ] `GET /api/moderation/logs`（分页筛选）

## E3. 接口质量要求
- [ ] 幂等性（重复点击提交不会重复创建）
- [ ] 并发安全（状态冲突返回业务码）
- [ ] 参数校验（长度、类型、枚举）

- [ ] 可疑账号冻结策略
- [ ] 管理员操作全审计

完成标准：

## J. 测试与质量保障（必须）

### J1. 缺口

### J2. 必做

- [ ] 回归用例：举报、评价、状态流转、并发加入

## K. 部署、运维、可观测（必须）

### K1. 缺口
- 无容器化、无 CI/CD、无监控、无备份

### K2. 必做

- [ ] 日志：结构化日志（请求ID、用户ID、耗时、错误码）
- [ ] 监控：健康检查、错误率、接口耗时

完成标准：

## L. 前端联调改造缺口（非常关键）

- [ ] 把本地 store 主数据源切到真实 API 拉取（首页、详情、任务、搭子、管理员）
- [ ] 把当前“先改本地再请求”改为“请求成功后更新状态”
- [ ] 全局错误处理与重试提示
- [ ] 空态/加载态/失败态统一
- [ ] token 持久化与过期跳转

完成标准：
- 关掉 `VITE_USE_MOCK_API` 后主流程全部可用

## 6. 一句话最终判定

目前项目距离“完整可用 APP”仍有系统性差距，核心不是“接口数量不够”，而是**后端真实业务化、数据库一致性、AI服务化、安全测试运维体系尚未建立**；按本清单补齐后，才能从演示系统转入可用系统。
